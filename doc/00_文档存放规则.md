> **版本**: 1.0.0
> **更新日期**: 2026-04-23

## 三、文件命名规则

### 3.1 标准命名格式

```
[类型描述]_[日期].md
```

- **日期格式**: `YYYY-MM-DD`（年-月-日）
- **日期位置**: 文件名末尾，描述之后
- **分隔符**: 使用下划线 `_` 而非空格或连字符

### 3.2 日期规则

| 文档类型 | 是否强制日期 | 日期含义 |
|----------|-------------|----------|
| 架构设计文档 | ❌ 可选 | 技术方案通常不随时间变化 |
| 产品设计文档 | ❌ 可选 | 交互设计通常不随时间变化 |
| **报告/审计文档** | ✅ **必须** | 报告必须注明生成日期 |
| 计划文档 | ✅ 必须 | 计划必须注明制定日期 |
| 会议记录 | ✅ 必须 | 会议必须注明日期 |

### 3.3 版本管理

文档内容变更时，有两种处理方式：

| 方式 | 适用场景 | 操作 |
|------|----------|------|
| **内容更新** | 活跃维护的文档 | 直接修改文件，更新顶部的 `版本` 和 `更新时间` |
| **版本归档** | 需要保留历史 | 旧版本移至 `archive/`，创建新文件 |

### 3.4 文档头部元数据

每个文档应包含标准头部：

```markdown
# 文档标题

> **版本**: X.Y.Z
> **更新日期**: YYYY-MM-DD
> **文档编辑**: 作者名
> **状态**: [草案/评审中/已发布/已归档]
> **关联文档**: [相关文档](路径)
```

## 四、AI 写入文档时的操作流程

当 AI 被要求"写一个文档"时，应按以下步骤操作：

### Step 1: 读取存放规则

```python
# AI 应该首先读取本文件
读取 doc/00_文档存放规则.md
```

### Step 2: 判断文档类型

根据用户描述判断文档属于哪个类别：

| 用户请求关键词 | 存放目录 |
|----------------|----------|
| "代码报告"、"审计"、"review" | `03_reports/` |
| "工作计划"、"todo"、"后续工作" | `04_plans/` |
| "架构设计"、"技术方案" | `01_architecture/` |
| "交互设计"、"用户体验" | `02_product/` |
| "会议纪要"、"决策" | `05_meetings/` |

### Step 3: 确定文件名

```
[描述]_[YYYY-MM-DD].md
```

示例:
- `代码报告_2026-04-23.md`
- `工作计划_2026-04-23.md`
- `安全审计_2026-04-23.md`

### Step 4: 检查是否需要归档旧版本

如果同名文件已存在且需要保留历史：
1. 将旧文件移动到 `archive/年份/`
2. 文件名追加 `_归档_YYYY-MM-DD.md`
3. 创建新文件

### Step 5: 写入文档内容

按照标准头部格式写入文档。

### Step 6: 更新索引

更新 `doc/INDEX.md`，添加新文档的链接。

## 五、日期辅助管理规则

### 5.1 快速识别最新文档

文件名包含日期，可通过排序快速识别：
```bash
ls -lt doc/03_reports/      # 按修改时间倒序
ls doc/03_reports/ | sort    # 按文件名字典序（日期在前）
```

### 5.2 优先级标记

在 `INDEX.md` 中，用 emoji 标记文档优先级：

| 标记 | 含义 |
|------|------|
| 🔴 | 高优先级 — 需要立即处理或关注 |
| 🟡 | 中优先级 — 待处理或需要关注 |
| 🟢 | 已完成 — 状态正常 |
| 📦 | 归档 — 历史参考 |

### 5.3 文档状态

在头部元数据中标注状态：

| 状态 | 说明 | 适用场景 |
|------|------|----------|
| `草案` | 初稿，待评审 | 新创建的文档 |
| `评审中` | 正在评审讨论 | 待确定的方案 |
| `已发布` | 正式发布 | 稳定的文档 |
| `已归档` | 已过期/废弃 | 移至 archive/ |

## 六、特殊规则

### 6.1 demo_doc/ 目录保留

`demo_doc/` 存放早期设计文档和 HTML 原型，**暂不动迁**。这些文档是历史参考，新文档不再放入此目录。

### 6.2 dev_doc/ 目录用途

`dev_doc/` 用于存放**开发过程中的临时文档**（如 PR review、临时笔记），这些文档不需要长期维护。

### 6.3 文档过期策略

| 文档类型 | 过期条件 | 处理方式 |
|----------|----------|----------|
| 工作计划 | 完成或过期后 | 移至 `archive/`，创建新的计划文档 |
| 审计报告 | 新报告生成后 | 旧报告移至 `archive/` |
| 架构文档 | 技术方案变更后 | 更新内容，标记新版本 |

## 七、常见问题

### Q1: 文档应该放在 doc/ 还是 dev_doc/?

| 场景 | 位置 |
|------|------|
| 正式文档（架构、报告、计划） | `doc/` |
| 开发过程中的临时笔记 | `dev_doc/` |
| PR review、代码审查记录 | `dev_doc/` |
| 需要长期维护的文档 | `doc/` |
| 一次性生成的分析报告 | `doc/03_reports/` |

### Q2: 如何判断文档是否需要日期?

如果文档内容**随时间变化**或需要**区分版本**，则必须加日期：
- 报告类：✅ 必须加日期
- 计划类：✅ 必须加日期
- 架构/设计：❌ 通常不加（除非有明显版本变化）

### Q3: 文档修改时是更新原文件还是创建新文件?

| 场景 | 处理方式 |
|------|----------|
| 小幅修改、更新内容 | 直接修改，更新版本号和日期 |
| 大幅变更、需要保留历史 | 旧文件归档，创建新文件 |
| 定期报告（如周报、月报） | 每次创建新文件 |
