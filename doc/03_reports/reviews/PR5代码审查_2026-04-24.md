## 三、问题详情

### C-1: CSS 样式重复定义 (🔴 高)

**文件**: `frontend/css/components.css`, `frontend/css/layout.css`

以下类在两个 CSS 文件中均有定义且属性不同：
- `.like-button-wrapper`
- `.like-button` / `.like-button:hover` / `.like-button.liked`
- `.like-icon` / `.like-count`

**冲突示例**:

| 属性 | `components.css` | `layout.css` |
|------|-------------------|--------------|
| `.like-button` border | `1px solid var(--color-border-subtle)` | `2px solid rgba(102, 8, 116, 0.1)` |
| `.like-button:hover` | `background: var(--color-bg-tertiary)` | `transform: scale(1.05)` |
| `.like-button.liked` | 红色风格 (`#dc2626`) | 紫色风格 (`--color-accent-primary`) |

**建议**: 保留 `components.css` 中的定义（作为通用组件样式），删除 `layout.css` 中的重复。或在 `layout.css` 中仅保留页面特有的覆盖样式。

---

### C-2: `confirm()` 阻塞式弹窗 (🟡 中)

**文件**: `frontend/pages/create.html`

```javascript
if (confirm(APP_STATE.language === 'zh'
    ? '是否前往探索页面选择位置？'
    : 'Go to Explore page to select a location?')) {
    window.location.href = 'explore.html';
}
```

`confirm()` 会阻塞 UI 线程，在移动端体验尤其差，且无法自定义样式。

**建议**: 使用项目已有的 Toast 组件 + 模态框替代，或直接跳转并提供"返回"按钮。

---

### C-3: 点赞按钮事件绑定模式脆弱 (🟡 中)

**文件**: `frontend/pages/gallery.html`

`initLikeButton` 使用 `hasAttribute('data-listener-added')` 避免重复绑定，但依赖模块级变量 (`currentDesignId`, `isLiked`, `currentLikesCount`) 作为闭包状态。虽然功能正确，但每次打开新设计详情时，这些变量的值会静默改变，增加了维护风险。

**建议**: 考虑每次打开模态框时解绑旧事件、绑定新事件，或使用更明确的状态管理模式。

---

### C-4: 搜索输入框类名语义不当 (🟢 低)

**文件**: `frontend/pages/gallery.html`

```html
<input type="text" id="searchInput" class="text-area" style="min-height: 48px; resize: none;">
```

单行搜索输入框使用了 `text-area` 类，然后通过内联样式覆盖其多行文本行为。

**建议**: 创建专用的 `.search-input` 类。

---

### C-5: 位置选项硬编码 (🟢 低)

**文件**: `frontend/pages/gallery.html`

位置筛选器（图书馆、信息楼、能源楼等）硬编码为静态 HTML。如果未来位置数据从 API 动态获取，此处会与后端不同步。

**建议**: 确认位置列表是否固定。如果会变化，应从 API 动态渲染。

---

### C-6: 日期排序回退值 (🟢 低)

**文件**: `frontend/pages/gallery.html`

```javascript
filtered.sort((a, b) => new Date(b.created_at || 0) - new Date(a.created_at || 0));
```

`new Date(0)` 返回 1970-01-01，将无 `created_at` 的条目排到末尾。但格式错误的日期字符串会产生 `Invalid Date`，导致 NaN 比较，排序结果不可预测。

**建议**: 添加日期有效性验证，或在数据层保证 `created_at` 字段始终存在。
