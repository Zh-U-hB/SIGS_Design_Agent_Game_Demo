## 三、严重问题详情

### C-1: `emotion_tags` 类型不匹配

**文件**: `backend/services/agent_service.py`

**问题**: 函数签名声明 `emotion_tags: Dict`，内部使用 `emotion_tags.keys()`。但 `DesignInputRequest` schema 定义 `emotion_tags: list[str] | None`，数据库模型存储为 `dict | None` (JSONB)。

**影响**: 运行时 `AttributeError`，`list` 对象没有 `keys()` 方法。

**位置**:
```python
# agent_service.py:25
async def translate_to_design_spec(emotion_tags: Dict, user_input: str, location_label: str) -> str:

# agent_service.py:64
emotion_desc = ", ".join(emotion_tags.keys()) if emotion_tags else "无特定情绪"
```

**修复建议**: 统一类型定义。推荐将 `agent_service` 改为兼容 `list[str]`：
```python
async def translate_to_design_spec(emotion_tags: list[str] | None, user_input: str, location_label: str | None) -> str:
    ...
    emotion_desc = ", ".join(emotion_tags) if emotion_tags else "无特定情绪"
```

---

### C-2: `get_design_status` 触发副作用

**文件**: `backend/services/design_service.py`（约第 140-155 行）

**问题**: 每次客户端轮询设计状态时，如果图片已生成但 3D 模型尚未完成，会**重新发起**一个 3D 转换请求。这导致：
- 重复创建任务，浪费 API 调用额度
- 无去重机制，可能产生大量冗余任务
- GET/状态端点不应有副作用（违反 REST 语义）

**位置**:
```python
if not design.model_3d_url or design.model_3d_url == "生成中...":
    from .model3d_service import convert_to_3d
    conversion_result = await convert_to_3d(...)  # 每次轮询都触发！
```

**修复建议**: 在 `confirm_design` 中触发一次 3D 转换，将 task_id 存入数据库。`get_design_status` 只查询已有任务状态，不创建新任务。

---

### C-3: `total_count` 语义错误

**文件**: `backend/services/design_service.py`（约第 214 行）

**问题**: 原代码通过 `SELECT COUNT(*)` 查询总记录数，用于分页元数据。PR 改为 `len(designs)`，返回的是**当前页**的记录数而非总数。

**影响**: 客户端分页计算 `total_pages = total_count / page_size` 将始终得到 1 页，无法正确翻页。

**修复建议**: 恢复 count 查询：
```python
count_result = await db.execute(select(func.count()).select_from(Design))
total_count = count_result.scalar() or 0
```

---

### C-4: `confirm_design` 未持久化 3D 转换

**文件**: `backend/services/design_service.py`（约第 61-98 行）

**问题**: `confirm_design` 中调用 `generate_image()` 后，无论成功与否都设置 `design.model_3d_url = None` 并提交。3D 转换从未在此流程中发起，与 C-2 耦合形成逻辑断层。

**修复建议**: 在图片生成成功后，立即发起 3D 转换并存储 task_id：
```python
if image_result["success"]:
    design.generated_image = image_result.get("image_url", "生成中...")
    # 发起 3D 转换
    conversion_result = await convert_to_3d(...)
    design.model_3d_url = conversion_result.get("model_url") or "生成中..."
    status = "processing"
```

## 四、重要问题详情

### C-5: 错误信息泄露

**文件**: `image_service.py`、`model3d_service.py`

**问题**: 异常捕获中将 `str(e)` 和 `e.response.text` 直接返回给客户端，可能包含 API 密钥、内部 URL、堆栈信息等敏感内容。

**修复建议**: 返回通用错误消息，详细信息仅记录日志：
```python
return {"success": False, "message": "图片生成服务暂时不可用"}
logger.error(f"详细错误: {e}")
```

### C-6: 导入路径不一致

**文件**: `design_service.py`

**问题**: 将 `from backend.database.models import Design` 改为 `from database.models import Design`，这依赖于运行时的工作目录在 `backend/` 下，降低了模块可移植性。应保持与项目现有风格一致。

### C-7: 冗余函数级 import

**文件**: `design_service.py`（约第 134 行）

**问题**: `get_design_status` 内部 `from .model3d_service import convert_to_3d`，但文件顶部已导入。冗余且造成困惑。

### C-8: 3D 服务复用图片 API 配置

**文件**: `model3d_service.py`

**问题**: 3D 转换 API 使用 `settings.IMAGE_API_KEY` 和 `settings.IMAGE_API_URL`，这是图片生成的配置。3D 转换是不同服务，应有独立配置项（如 `MODEL3D_API_KEY`、`MODEL3D_API_URL`）。
