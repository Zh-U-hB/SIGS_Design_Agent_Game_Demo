## 三、安全漏洞审查

### 3.1 严重漏洞 (CRITICAL)

#### C1: API Key 时序攻击 — 认证绕过

- **文件**: `backend/middleware/auth.py:30`, `backend/api/deps.py:22`
- **问题**: 使用 `!=` 进行 API Key 比较，而非常量时间比较函数
- **攻击方式**: 攻击者通过测量响应时间逐字符猜测 API Key
- **修复**: 使用 `hmac.compare_digest(api_key, settings.API_KEY)`

#### C2: 点赞计数竞态条件 — 数据不一致

- **文件**: `backend/services/like_service.py:36-38, 69-71`
- **问题**: `likes_count` 在 Python 层面读取-修改-写回，无行级锁
- **影响**: 1000 并发下计数将严重偏离实际点赞数
- **修复**: 使用 SQL 原子更新 `UPDATE designs SET likes_count = likes_count + 1 WHERE id = :id`

#### C3: 前端 XSS — innerHTML 未过滤

- **文件**: `frontend/pages/generate.html:219`, `frontend/js/utils.js:79-188`
- **问题**: 使用 `innerHTML` 渲染 API 返回数据，未经转义
- **攻击方式**: 通过数据库存储恶意内容（Stored XSS）或 MITM 篡改响应
- **修复**: 使用 `textContent` 或 DOMPurify 进行消毒

### 3.2 高危漏洞 (HIGH)

#### H1: 点赞接口会话伪造

- **文件**: `backend/api/design.py:78-103`
- **问题**: `session_id` 作为查询参数传入，无所有权验证；任何认证用户可代替其他用户点赞
- **额外风险**: `session_id` 出现在 URL 中（浏览器历史、日志、Referer 头）

#### H2: 开发模式静默绕过认证

- **文件**: `backend/middleware/auth.py:26-27`
- **问题**: `API_KEY` 为空时所有请求自动放行，无日志警告
- **影响**: 生产环境配置错误时完全无认证

#### H3: 设计接口 IDOR

- **文件**: `backend/services/design_service.py:82-108`
- **问题**: 任何认证用户可通过 UUID 获取任意设计详情，无所有权检查

#### H4: 前端 API Key 存储在 localStorage

- **文件**: `frontend/js/config.js:9, 78`
- **问题**: API Key 存于 localStorage，XSS 可窃取；虽有前端 JS 但当前已注释未启用

#### H5: 无 CSRF 保护

- **文件**: `frontend/js/api.js:19-50`
- **问题**: 所有状态变更请求无 CSRF Token

#### H6: 前端认证代码已注释

- **文件**: `frontend/js/api.js:28-31`
- **问题**: `X-API-Key` 头设置代码被注释，所有 API 请求不带认证

## 四、并发与性能审查（1000 并发目标）

### 4.1 当前瓶颈

| 瓶颈 | 当前值 | 建议值 | 文件 |
|------|--------|--------|------|
| 数据库连接池 | `pool_size=5` | ≥ 20 | `database/connection.py:24` |
| 最大溢出连接 | `max_overflow=10` | ≥ 30 | `database/connection.py:24` |
| Uvicorn workers | 1（默认） | 4-8（按 CPU 核数） | 启动命令 |
| 无连接池超时 | 未设置 | `pool_timeout=30` | `database/connection.py` |
| 无连接回收 | 未设置 | `pool_recycle=1800` | `database/connection.py` |

### 4.2 性能问题

| 问题 | 影响 | 文件 |
|------|------|------|
| `BaseHTTPMiddleware` 性能开销 | 每个 HTTP 请求额外创建 asyncio task，高并发下开销显著 | `middleware/auth.py`, `middleware/logging.py` |
| `get_map_points` 无分页 | 返回全部位置数据，数据量大时内存飙升 | `services/design_service.py:184-202` |
| 统计查询全表加载 | `locations_covered` 在 Python 层计算去重，应在 SQL 层用 `COUNT(DISTINCT)` | `api/stats.py:37-48` |
| 会话永不过期 | `users` 表无限增长，查询变慢 | `services/session_service.py` |
| 无请求体大小限制 | 攻击者可发送超大 JSON 耗尽内存 | 全局 |
| `likes.design_id` 无独立索引 | 按设计查点赞列表需全表扫描 | `database/models.py:46-55` |

### 4.3 并发承载能力评估

以当前配置（pool_size=5, 1 worker）评估：

- **数据库**: 5 个连接 × 1 worker = 最多 5 个并发数据库操作，其余排队等待
- **CPU**: 1 个 Uvicorn worker 处理所有请求，CPU 密集操作（如 JSON 解析）会成为瓶颈
- **内存**: 无会话过期 + 无请求体限制，内存使用不可控

**预计并发承载能力**: ~50-100 并发请求（远低于 1000 目标）

## 九、1000 并发承载路线图

要达到 1000 并发承载能力，需要以下架构调整：

1. **Uvicorn 多 worker**: `uvicorn main:app --workers 4-8`（按 CPU 核数）
2. **数据库连接池**: `pool_size=20, max_overflow=30, pool_timeout=30, pool_recycle=1800`
3. **反向代理**: Nginx 前置，处理 TLS 终止、静态文件、请求缓冲、速率限制
4. **Redis 缓存**: 缓存统计查询、地图数据（当前每次请求都查数据库）
5. **异步任务队列**: 图片生成、3D 模型生成等耗时操作改为后台任务（Celery/ARQ）
6. **CDN**: 静态资源（CSS/JS/图片）通过 CDN 分发
7. **数据库索引优化**: 为高频查询添加索引
8. **会话存储优化**: 考虑 Redis 存储会话，减轻数据库压力

**预估架构调整工作量**: 3-5 人天
