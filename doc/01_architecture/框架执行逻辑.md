## 一、总体架构概览

```
┌─────────────────────────────────────────────────────┐
│                    用户浏览器                         │
│                                                     │
│  landing → explore → create → confirm → generate    │
│                          ↓ 社区                      │
│                        gallery                       │
└────────────────────────┬────────────────────────────┘
                         │ HTTP 请求（REST API）
                         ↓
┌─────────────────────────────────────────────────────┐
│                 FastAPI 后端 (main.py)                │
│                                                     │
│  请求流向：                                           │
│  CORS中间件 → API Key认证 → 日志记录 → 路由匹配        │
│       → 依赖注入(DB会话) → 端点处理函数 → Service层    │
│       → 统一响应格式返回                               │
└──────────┬──────────────────────┬───────────────────┘
           │                      │
           ↓                      ↓
┌──────────────────┐   ┌─────────────────────┐
│  PostgreSQL 数据库 │   │  外部 AI 服务（占位）  │
│  (connection.py)  │   │  Agent / 图生图 / 3D  │
└──────────────────┘   └─────────────────────┘
```

## 二、后端启动流程

入口文件：`backend/main.py`

```
uvicorn main:app 启动
    │
    ├── 1. 加载 config.py
    │       └── Settings 类从 .env 读取环境变量
    │           → DATABASE_URL, API_KEY, FRONTEND_URL 等
    │
    ├── 2. 创建 FastAPI 实例 (main.py:31)
    │       └── 绑定 lifespan 生命周期（启动/关闭日志）
    │
    ├── 3. 注册中间件（按添加顺序的反序执行，即最先生效的是最后添加的）
    │       ├── CORS 中间件 ← 使用 settings.FRONTEND_URL
    │       ├── APIKeyMiddleware (middleware/auth.py) ← 使用 settings.API_KEY
    │       └── LoggingMiddleware (middleware/logging.py) ← 使用 utils/logger.py
    │
    ├── 4. 注册异常处理器 (middleware/error_handler.py)
    │       ├── RequestValidationError → 错误码 40001
    │       └── Exception → 错误码 50001
    │
    ├── 5. 挂载 API 路由 (api/router.py)
    │       └── 前缀 /api/v1
    │           ├── session_router  (api/session.py)
    │           ├── design_router   (api/design.py)
    │           └── stats_router    (api/stats.py)
    │
    └── 6. 挂载前端静态文件
            └── frontend/ 目录 → 通过 StaticFiles 提供
```

## 三、单个请求的完整处理流程

以 `POST /api/v1/designs/input` 为例：

```
用户点击"让AI来理解我的想法"
    │
    ↓ 前端发起请求
frontend/js/api.js: apiPost("/designs/input", {body})
    │  └── apiRequest() → fetch() → 添加 X-API-Key 头
    │
    ↓ HTTP 请求到达后端
    │
┌─── 后端处理开始 ────────────────────────────────────────┐
│                                                         │
│  1. CORS 中间件                                         │
│     检查请求来源是否在允许列表中                            │
│     文件：main.py:38 (CORSMiddleware)                    │
│                                                         │
│  2. API Key 认证中间件                                   │
│     检查 X-API-Key 请求头是否匹配 settings.API_KEY       │
│     不匹配 → 直接返回 401 + 错误码 40101                  │
│     文件：middleware/auth.py:17 (dispatch方法)            │
│     使用：config.py 的 settings, schemas/common.py 的 api_error()  │
│                                                         │
│  3. 请求日志中间件                                       │
│     记录请求方法、路径，计算响应耗时                        │
│     文件：middleware/logging.py:17 (dispatch方法)        │
│     使用：utils/logger.py 的 get_logger()                │
│                                                         │
│  4. 路由匹配                                            │
│     /api/v1/designs/input → design_router               │
│     文件：api/router.py:10 → api/design.py:15            │
│                                                         │
│  5. 依赖注入                                            │
│     get_db() → get_db_session() → _get_session_factory() │
│     → _get_engine() → 创建 AsyncSession                  │
│     文件：api/deps.py:14 → database/connection.py:30     │
│                                                         │
│  6. 请求体验证                                          │
│     自动用 DesignInputRequest (Pydantic模型) 解析请求体   │
│     验证失败 → RequestValidationError → 错误码 40001     │
│     文件：schemas/design.py:9 (DesignInputRequest)       │
│                                                         │
│  7. 端点处理函数执行                                     │
│     submit_creative_input(request, db)                   │
│     文件：api/design.py:15                               │
│     调用 → design_service.submit_input(db, request)      │
│     文件：services/design_service.py:9                   │
│                                                         │
│  8. Service 层（当前为占位）                              │
│     TODO: 查询用户、创建记录、调用 Agent 服务             │
│     raise NotImplementedError                           │
│                                                         │
│  9. 统一响应格式                                        │
│     api_success(data=result)                             │
│     → {"code": 0, "message": "success", "data": {...}}  │
│     文件：schemas/common.py:40                           │
│                                                         │
└─── 后端处理结束 ────────────────────────────────────────┘
    │
    ↓ 前端接收响应
frontend/js/api.js: apiRequest() 解析响应
    │  └── 检查 data.code === 0
    │  └── 返回 data 给调用方
    │
    ↓ 前端后续处理
setState("designId", result.data.id)
navigateTo(PAGES.CONFIRM) → 跳转到确认页
```

## 四、后端各层文件职责与调用关系

### 4.1 配置层

| 文件 | 职责 | 被谁使用 |
|------|------|---------|
| `config.py` | 从 .env 读取所有环境变量 | main.py, auth.py, connection.py, deps.py |

```
config.py (Settings 类)
    │
    ├── settings.DATABASE_URL → database/connection.py:18 (创建引擎)
    ├── settings.API_KEY      → middleware/auth.py:25 (认证校验)
    └── settings.FRONTEND_URL → main.py:40 (CORS 白名单)
```

### 4.2 中间件层

| 文件 | 职责 | 执行时机 |
|------|------|---------|
| `middleware/auth.py` | API Key 认证 | 每个 `/api/v1/*` 请求 |
| `middleware/logging.py` | 请求日志 | 每个请求 |
| `middleware/error_handler.py` | 全局异常捕获 | 发生异常时 |

```
请求进入
    │
    ├── auth.py: APIKeyMiddleware.dispatch()
    │   ├── 路径不含 /api/v1/ → 放行
    │   ├── X-API-Key 匹配 → 放行
    │   └── 不匹配 → 返回 {code: 40101, message: "API Key 缺失或无效"}
    │
    ├── logging.py: LoggingMiddleware.dispatch()
    │   └── 记录: "GET /api/v1/designs -> 200 (0.05s)"
    │
    └── error_handler.py: add_error_handlers()
        ├── RequestValidationError → {code: 40001}
        └── Exception → {code: 50001} + 写日志
```

### 4.3 路由层（API 端点）

| 文件 | 端点 | 调用的 Service |
|------|------|---------------|
| `api/session.py` | `POST /sessions` | `session_service.create_session()` |
| `api/design.py` | `POST /designs/input` | `design_service.submit_input()` |
| `api/design.py` | `POST /designs/confirm` | `design_service.confirm_design()` |
| `api/design.py` | `GET /designs/{id}` | `design_service.get_design()` |
| `api/design.py` | `GET /designs/{id}/status` | `design_service.get_design_status()` |
| `api/design.py` | `GET /designs` | `design_service.list_designs()` |
| `api/design.py` | `GET /designs/map` | `design_service.get_map_points()` |
| `api/design.py` | `POST /designs/{id}/like` | `session_service.get_session()` → `like_service.like_design()` |
| `api/design.py` | `DELETE /designs/{id}/like` | `session_service.get_session()` → `like_service.unlike_design()` |
| `api/stats.py` | `GET /stats` | 无（内联返回占位数据） |

### 4.4 依赖注入层

| 文件 | 提供的依赖 | 使用位置 |
|------|-----------|---------|
| `api/deps.py` → `get_db()` | 数据库会话 (AsyncSession) | 所有端点的 `Depends(get_db)` |
| `api/deps.py` → `verify_api_key()` | API Key 验证 | 备用（当前由中间件统一处理） |
| `api/deps.py` → `get_current_session()` | 通过 session_id 获取用户 | 备用（当前各端点自行调用） |

```
get_db()                          [api/deps.py:14]
  └── get_db_session()            [database/connection.py:30]
        └── _get_session_factory() [database/connection.py:23]
              └── _get_engine()    [database/connection.py:14]
                    └── create_async_engine(settings.DATABASE_URL)
```

### 4.5 Schema 层（数据验证模型）

```
schemas/common.py ─── 全局使用
  ├── 错误码常量: SUCCESS, BAD_PARAMS, AUTH_ERROR, NOT_FOUND, SERVER_ERROR
  ├── api_success(data)    → 所有端点的成功响应
  └── api_error(code, msg) → 错误处理器的错误响应

schemas/session.py ─── api/session.py 使用
  ├── SessionCreateRequest  → 请求体模型（空）
  └── SessionResponse       → 响应数据模型

schemas/design.py ─── api/design.py 使用
  ├── DesignInputRequest     → POST /designs/input 的请求体
  ├── DesignConfirmRequest   → POST /designs/confirm 的请求体
  ├── DesignResponse         → GET /designs/{id} 的响应数据
  ├── DesignStatusResponse   → GET /designs/{id}/status 的响应数据
  ├── DesignMapPoint         → GET /designs/map 的响应数据项
  └── DesignListParams       → GET /designs 的查询参数

schemas/stats.py ─── api/stats.py 使用
  └── StatsResponse → GET /stats 的响应数据
```

### 4.6 Service 层（业务逻辑占位）

```
services/
  │
  ├── session_service.py      ← 被 api/session.py, api/design.py 调用
  │   ├── create_session(db)         → 创建访客会话
  │   └── get_session(db, session_id) → 查询会话用户
  │
  ├── design_service.py       ← 被 api/design.py 调用
  │   ├── submit_input(db, request)      → 核心：接收创意 → 调用 Agent
  │   ├── confirm_design(db, request)    → 确认设计 → 触发图生图
  │   ├── get_design(db, design_id)      → 查询设计详情
  │   ├── get_design_status(db, id)      → 查询生成状态
  │   ├── list_designs(db, params)       → 分页列表
  │   └── get_map_points(db)             → 地图光点数据
  │
  ├── agent_service.py        ← 被 design_service.py 调用
  │   └── translate_to_design_spec(tags, input, label)
  │       → 调用 LLM API，翻译非专业描述为设计说明
  │
  ├── image_service.py        ← 被 design_service.py 调用
  │   ├── generate_image(screenshot_url, description)
  │   │   → 调用图生图 API（Stable Diffusion 等）
  │   └── check_generation_status(task_id)
  │       → 查询异步生成任务状态
  │
  ├── model3d_service.py      ← 被 design_service.py 调用
  │   └── convert_to_3d(image_url)
  │       → 调用图转3D API（Hunyuan3D 等）
  │
  └── like_service.py         ← 被 api/design.py 调用
      ├── like_design(db, user_id, design_id)   → 点赞
      └── unlike_design(db, user_id, design_id)  → 取消点赞
```

### 4.7 Service 之间的设计流程调用链

这是完整的六阶段设计流程中，Service 层的调用关系：

```
阶段三：创意输入
  design_service.submit_input()
    ├── 1. session_service.get_session(db, session_id)     → 获取/创建用户
    ├── 2. 创建 Design 记录（位置、情绪标签、文字）
    ├── 3. agent_service.translate_to_design_spec()        → AI Agent 处理
    │       └── 调用外部 LLM API
    └── 4. 更新 Design 记录（写入 design_description 和 ai_response）

阶段四：确认设计
  design_service.confirm_design()
    ├── 1. session_service.get_session(db, session_id)     → 验证用户
    ├── 2. 更新 Design 记录（用户确认/编辑后的描述）
    └── 3. image_service.generate_image()                  → 触发图生图
            └── 调用外部图生图 API

阶段五：生成结果（异步）
  design_service.get_design_status()                       → 前端轮询
    └── 返回 status: "processing" | "completed" | "failed"

  （异步后台任务，完成后自动执行）
  model3d_service.convert_to_3d()                          → 图转3D模型
    └── 调用外部图转3D API
```

## 七、队友实现指南

### 需要实现的 Service 文件（按优先级排序）

| 优先级 | 文件 | 原因 |
|--------|------|------|
| 1 | `services/session_service.py` | 所有其他服务都依赖用户身份 |
| 2 | `services/design_service.py` | 核心设计流程，串联所有其他服务 |
| 3 | `services/agent_service.py` | 被设计流程调用，需要对接 LLM API |
| 4 | `services/image_service.py` | 被设计流程调用，需要对接图生图 API |
| 5 | `services/like_service.py` | 社区功能，独立模块 |
| 6 | `services/model3d_service.py` | 异步后台任务，可最后实现 |

### 每个占位函数的 TODO 说明

每个 Service 文件中的函数都有 docstring，描述了具体实现步骤。例如 `design_service.submit_input()` 的 TODO：

```
1. 通过 session_id 查找或创建用户       → 调用 session_service
2. 创建设计记录（位置+情绪标签+用户输入） → 写入 designs 表
3. 调用 agent_service.translate_to_design_spec() → AI Agent 处理
4. 更新设计记录中的 AI 回应              → 更新 designs 表
5. 返回包含 ai_response 的设计数据
```
