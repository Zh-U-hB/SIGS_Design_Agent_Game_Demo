## Git 版本管理规则

### 分支策略

- `main`：开发分支，日常开发与功能迭代在此分支进行
- `serve`：稳定分支，经过测试验证的稳定版本合并至此，用于部署
- `feature/xxx`：功能分支（可选），大型功能开发时从 main 拉取，完成后合并回 main

### Commit 规范

提交信息格式：`<类型>: <简述>`

类型包括：
- `feat`: 新功能
- `fix`: 修复 Bug
- `refactor`: 重构（不改变功能）
- `test`: 添加或修改测试
- `docs`: 文档更新
- `style`: 代码格式调整（不影响逻辑）
- `chore`: 构建、依赖、配置等杂项

示例：`feat: 添加用户注册接口`

### 合并规则

- 功能开发完成后需通过对应测试才能合并
- 合并到 `serve` 分支前需确认所有测试通过
- 禁止使用 `--force` 推送到 `serve` 分支

## 代码开发规则

### 后端规则

- 开发语言：Python
- 包管理：使用 uv 管理虚拟环境和依赖
- 如果项目中没有 `.venv` 虚拟环境目录，则应先创建并执行 `uv sync` 同步依赖
- Web 框架：使用 FastAPI
- ORM/数据库驱动：使用 SQLAlchemy + asyncpg（异步 PostgreSQL 驱动）

### 前端规则

- 实现方式：原生 HTML / CSS / JavaScript
- 包管理：npm
- 前后端通过 RESTful API 接口通讯
- 可通过内网穿透发送到公网进行访问
- 前端静态文件由后端 FastAPI 提供服务或独立部署

### 通用代码规范

- 高内聚、低耦合原则
- 单个函数不超过 200 行
- 单个代码文件不超过 800 行
- 各功能代码按分类存放到对应目录下

### 命名规范

| 范围 | 风格 | 示例 |
|------|------|------|
| Python 文件名 | snake_case | `user_service.py` |
| Python 变量/函数 | snake_case | `get_user_by_id` |
| Python 类名 | PascalCase | `UserService` |
| Python 常量 | UPPER_SNAKE_CASE | `MAX_RETRY_COUNT` |
| HTML/CSS/JS 文件名 | kebab-case | `game-board.html` |
| CSS 类名 | kebab-case | `game-container` |
| JS 变量/函数 | camelCase | `getUserData` |
| JS 常量 | UPPER_SNAKE_CASE | `API_BASE_URL` |
| API 路径 | kebab-case 小写复数 | `/api/v1/users` |

### 格式检查

后端使用 Ruff 进行代码格式化和静态检查：

```bash
# 检查格式问题
ruff check .

# 自动修复可修复的问题
ruff check --fix .

# 格式化代码
ruff format .
```

代码编写完成后必须先通过 Ruff 检查，存在问题则先修复后再提交。

### 测试规范

一个功能开发完成后必须在 `test/` 目录下编写对应的功能测试脚本，测试通过才算功能开发完成。

测试脚本路径规则——与源码路径镜像：
- 源码：`backend/database/serve.py` → 测试：`test/backend/database/test_serve.py`
- 源码：`backend/api/user.py` → 测试：`test/backend/api/test_user.py`

一个代码文件对应一个测试脚本。

## API 接口规范

### 基本规则

- 风格：RESTful
- 数据格式：JSON
- 路径前缀：`/api/v1/`
- 使用复数名词表示资源集合：`/api/v1/users`、`/api/v1/games`

### HTTP 方法语义

| 方法 | 用途 | 示例 |
|------|------|------|
| GET | 获取资源 | `GET /api/v1/games` |
| POST | 创建资源 | `POST /api/v1/games` |
| PUT | 全量更新 | `PUT /api/v1/games/1` |
| PATCH | 部分更新 | `PATCH /api/v1/games/1` |
| DELETE | 删除资源 | `DELETE /api/v1/games/1` |

### 认证方式

使用 API Key 认证：
- 请求头携带：`X-API-Key: <your-api-key>`
- API Key 存储在环境变量中，禁止硬编码

### 统一响应格式

成功响应：
```json
{
  "code": 0,
  "message": "success",
  "data": { ... }
}
```

错误响应：
```json
{
  "code": 40001,
  "message": "错误描述",
  "data": null
}
```

常用错误码：
- `0`：成功
- `40001`：参数错误
- `40101`：API Key 缺失或无效
- `40401`：资源不存在
- `50001`：服务器内部错误

## 数据库管理规则

- 数据库类型：PostgreSQL（云端部署）
- 禁止随意删除、修改云端数据库的数据
- 数据库变更必须通过迁移脚本（migration）执行，禁止手动修改表结构
- 敏感操作（删表、清空数据）必须在迁移脚本中有明确注释说明原因

## 环境配置规则

- 所有环境变量必须记录在 `.env.example` 中（不含真实值），仅作为模板
- 新增环境变量时必须同步更新 `.env.example`
- `.env` 文件已在 `.gitignore` 中，禁止提交到仓库
- 敏感信息（数据库密码、API Key 等）一律通过环境变量注入，禁止硬编码

### 必要的环境变量

```
# 数据库配置
DATABASE_URL=
DATABASE_NAME=

# API 配置
API_KEY=
API_HOST=
API_PORT=

# 前端配置
FRONTEND_URL=
```

## 安全规范

- 所有外部输入必须校验后再使用（防止注入、XSS 等）
- SQL 查询使用参数化查询，禁止拼接 SQL 字符串
- API Key 等敏感信息禁止出现在日志输出中
- 生产环境必须启用 HTTPS
- 错误响应不得泄露内部实现细节（如堆栈信息、SQL 语句）

## 错误处理与日志

### 错误处理

- 后端使用 FastAPI 的异常处理机制统一捕获和返回错误
- 不允许裸 `except`，必须指定异常类型
- 关键操作失败时记录日志，不静默忽略异常

### 日志规范

- 使用 Python 标准 `logging` 模块
- 日志级别：DEBUG < INFO < WARNING < ERROR < CRITICAL
- 日志内容应包含时间、级别、模块、具体信息
- 日志文件存放在 `temp/logs/` 目录下

## temp 目录管理

- `temp/` 用于存放临时文件，已加入 `.gitignore`，不会被提交
- `temp/logs/`：运行日志
- `temp/cache/`：临时缓存
- `temp/` 下的文件可随时清理，不应存放重要数据

## 部署规则

### 后端启动

```bash
cd backend
uv run uvicorn main:app --host 0.0.0.0 --port 8888 --reload
```

### 前端访问

- 开发环境：直接用浏览器打开 HTML 文件，或通过后端静态文件服务访问
- 生产环境：配置 Nginx 反向代理，将前端请求指向静态文件目录，API 请求转发至后端

### 内网穿透

如需公网访问，使用内网穿透工具将后端端口映射到公网地址。

## 文档管理规则

### 文档目录结构

项目文档按分类存放在 `doc/` 目录下：

```
doc/
├── 00_文档存放规则.md          # 文档存放规则定义（AI 写文档前必读）
├── INDEX.md                     # 主索引 — 所有文档的导航目录
│
├── 01_architecture/             # 架构设计 — 技术架构、系统设计、数据库设计
├── 02_product/                  # 产品设计 — 交互设计、功能需求、用户流程
├── 03_reports/                  # 报告文档 — 代码报告、审计报告、项目复盘
├── 04_plans/                    # 计划文档 — 工作计划、路线图、待办事项
├── 05_meetings/                 # 会议记录 — 会议纪要、决策记录
├── 06_references/               # 参考资料 — 外部文档、研究资料
└── archive/                     # 归档区 — 过期/废弃文档
```

### AI 写文档规范

当 AI 被要求"写一个文档"或"生成报告"时，必须按以下步骤操作：

1. **读取存放规则**：首先读取 `doc/00_文档存放规则.md`
2. **判断文档类型**：根据用户描述选择对应目录
   - 报告/审计 → `doc/03_reports/`
   - 工作计划 → `doc/04_plans/`
   - 架构设计 → `doc/01_architecture/`
   - 产品设计 → `doc/02_product/`
   - 会议记录 → `doc/05_meetings/`
3. **使用标准命名**：`[描述]_[YYYY-MM-DD].md`（报告和计划必须包含日期）
4. **包含标准头部**：
   ```markdown
   # 文档标题

   > **版本**: X.Y.Z
   > **更新日期**: YYYY-MM-DD
   > **文档编辑**: 作者名
   > **状态**: [草案/评审中/已发布/已归档]
   > **关联文档**: [相关文档](路径)
   ```
5. **更新索引**：在 `doc/INDEX.md` 中添加新文档链接

### 代码中的文档注释

- Python：使用 docstring 描述函数/类功能
- JavaScript：使用 JSDoc 风格注释
- 复杂逻辑必须添加注释说明

## AI Agent 辅助文件

项目包含多个针对不同 AI 编程助手的配置文件：

| 文件 | 适用工具 | 说明 |

1. 当 AI Agent 首次参与项目时，必须先阅读 `开发规则.md`
2. 各 Agent 特定文件是 `开发规则.md` 的补充，针对特定工具优化
3. 规则冲突时优先级：`开发规则.md` > `AGENT.md` > 工具特定文件 > 其他文档
4. 如果 `开发规则.md` 更新，AI Agent 应同步更新其对应的配置文件
